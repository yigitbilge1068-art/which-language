========================================
1. SYSTEM OVERVIEW
========================================
mini-scoreboard is a robust, command-line based sports data management system.
It features defensive programming principles to prevent data corruption.

========================================
2. DATA ARCHITECTURE & SCHEMA
========================================
Storage Path: .miniscore/matches.dat
Delimiter: | (Pipe character)
Format: ID|League|Week|Date|TeamA|ScoreA|TeamB|ScoreB

*** DATA INTEGRITY & SECURITY RULES (CRITICAL) ***
1. Delimiter Protection: The pipe character '|' is STRICTLY FORBIDDEN in any user input.
2. Type Validation: ScoreA and ScoreB MUST be positive integers (digits only).
3. Ghost Teams: Team names cannot be empty or just whitespaces.
4. Clone Matches: A team cannot play against itself (TeamA != TeamB).
5. Date Format: Dates must strictly be 10 characters long with two dashes (YYYY-MM-DD).
6. Length Limits: League, Week, Date, and Team names cannot exceed 30 characters to prevent buffer issues.
7. Case Insensitivity: Team names are automatically stripped of spaces and converted to UPPERCASE.

========================================
3. COMMAND SPECIFICATIONS
========================================
> init
Usage: python miniscore.py init
Description: Safely initializes the system (Checks both directory and file).

Usage: python miniscore.py add-match <League> <Week> <Date> <TeamA> <ScoreA> <TeamB> <ScoreB>
Description: Validates inputs strictly against the Security Rules before writing to disk.
Success: "Added match #<ID>: [<League> - <Week>] <TeamA> <ScoreA> - <ScoreB> <TeamB> on <Date>"

========================================
5. v0 LIMITATIONS & TECHNICAL DEBT (KNOWN ISSUES)
========================================
Since v0 strictly prohibits the use of loops (for/while) and lists ([]), the current architecture has the following known technical debts, which will be refactored in future weeks:

1. Fragile ID Generation: 
   - Current State: Match IDs are generated by counting newline characters (\n) in the file.
   - Vulnerability: If a user manually edits matches.dat and leaves an empty line, the ID generation will break (creating duplicate IDs).
   - Future Fix: Will use loops to read the file line-by-line, parse the highest ID, and increment it.

2. Basic Date Validation:
   - Current State: Date is checked by length (10 chars) and specific index dashes (YYYY-MM-DD).
   - Vulnerability: Cannot mathematically verify if the month is valid (e.g., 99 is accepted) or leap years.
   - Future Fix: Will implement proper string parsing/splitting when lists are allowed, or use the datetime module.

3. Lack of Database Locking (Race Condition):
   - Current State: Read/Write operations happen sequentially without locking.
   - Vulnerability: If two commands run at the exact same millisecond, data corruption may occur.
   - Future Fix: Out of scope for a basic CLI tool, but noted for architectural awareness.
